## 1. Описание задачи
Необходимо перемножить 2 квадратные матрицы размера 4096x4096 с элементами типа `float`. 
Исходные матрицы генерируются случайным образом в диапазоне от -1.0 до 1.0 и сохраняются в бинарные файлы с помощью вспомогательной программы `MatrixGenerator`.

Для оценки эффективности алгоритмов используются следующие формулы:
* **Сложность алгоритма (количество операций):** $c = 2n^3$, где $n$ - размерность матрицы (4096).
* **Производительность (MFlops):** $p = \frac{c}{t} \cdot 10^{-6}$, где $t$ - время работы алгоритма в секундах.

В работе реализовано и проанализировано 3 варианта перемножения матриц.

## 2. Инструкция по сборке проекта
Проект настроен с использованием системы сборки CMake и является кроссплатформенным/автономным. В папке `lib` уже находится скомпилированная библиотека OpenBLAS для Windows-систем.

**Сборка через консоль:**
1. `cmake -B build -DCMAKE_BUILD_TYPE=Release`
2. `cmake --build build --config Release`
3. Сначала запустите `build/MatrixGenerator.exe` для создания файлов матриц `matrix_A.bin` и `matrix_B.bin`.
4. Затем запустите `build/calculator.exe` для выполнения вычислений.

## 3. Анализ алгоритмов перемножения

### Вариант 1: Наивный алгоритм
Стандартный алгоритм перемножения матриц с использованием трех вложенных циклов.
Алгоритм имеет вычислительную сложность $O(N^3)$ и абсолютно не учитывает архитектуру кэш-памяти процессора.
* **Результат:** Пропущен при тестировании больших матриц. Из-за неоптимального доступа к памяти выполняется очень долго.

### Вариант 2: Использование библиотеки BLAS
Для эталонного тестирования используется функция `cblas_sgemm` из библиотеки OpenBLAS.
* **Время выполнения:** ~3.6742 с. 
* **Производительность:** ~37406 MFlops. 

### Вариант 3: Оптимизированный алгоритм
Собственный оптимизированный алгоритм был спроектирован с учетом возможностей современных процессоров. Применены следующие техники:
1. **Изменение порядка циклов (i-k-j):** Обеспечивает последовательный доступ к элементам матриц, что значительно снижает количество кэш-промахов.
2. **Блочное умножение (Tiling):** Матрицы разбиты на блоки размером 64x64 элемента (16 Кбайт). Это позволяет данным полностью помещаться в сверхбыстрый кэш L1 процессора (обычно 32-48 Кбайт на ядро).
3. **Параллелизм:** Использована технология OpenMP (`#pragma omp parallel for collapse(2)`) для распределения вычислений независимых блоков между ядрами процессора.
4. **Векторизация (SIMD):** Использована директива `#pragma omp simd` и флаг `-march=native` для задействования широких векторных регистров (AVX2/AVX-512) процессора.

* **Время выполнения:** ~6.2141 с. 
* **Производительность:** ~22117 MFlops. 

## 4. Оценка точности и компенсация погрешностей
Для проверки корректности алгоритма использовалась относительная норма Фробениуса между эталонной матрицей и матрицей, полученной собственным алгоритмом.

Относительная ошибка составила **`1e-6`**.
Такая высокая точность для типа `float` при $16.7$ миллионах сложений для каждого элемента достигается за счет применения аппаратного слияния операций **FMA (Fused Multiply-Add)**. Компилятору с помощью флага `-ffast-math` было разрешено выполнять умножение и сложение за один процессорный такт, что привело к выполнению только **одного** математического округления вместо двух на каждом шаге. Это предотвратило накопление машинной погрешности.

## 5. Выводы
Требование о достижении производительности не ниже 30% от библиотечной функции BLAS успешно выполнено.
Собственный оптимизированный алгоритм достиг производительности **~59.1%** от OpenBLAS (22117 MFlops против 37406 MFlops у эталона). Высокая производительность и точность были обеспечены правильной работой с подсистемой памяти (tiling, кэш L1) и аппаратными возможностями процессора (OpenMP, SIMD, FMA).
