## Главное изменение, которое помогло добиться такой скорости **в моём случае**: упаковка данных против Cache/TLB Thrashing

Поскольку матрицы размером 4096 x 4096 вызывают конфликты в кэше и TLB из-за того, что шаг в памяти является точной степенью двойки ($2^{12}$), нужно разорвать эту зависимость.
Нельзя просто изменить N, так как это сломает чтение из бинарного файла. Применим технику **Data Packing**. Будем копировать блок матрицы B в маленький локальный массив, который лежит в памяти непрерывно, и умножать на него.

   - **Источник:** Kazushige Goto, Robert A. van de Geijn - "*Anatomy of High-Performance Matrix Multiplication*" (ACM Transactions on Mathematical Software, 2008). Это фундаментальная статья, на которой построена библиотека OpenBLAS.
   - **Где искать:** Глава 4. "*Packing*", секция *TLB Misses*.

   - **Суть:** "*The TLB is a small cache... A TLB miss is costly. Packing a submatrix into a contiguous buffer in memory minimizes TLB misses and eliminates cache associativity conflicts.*". TLB - это маленький кэш... Промах в TLB обходится дорого. Упаковка подматрицы в непрерывный буфер в памяти минимизирует промахи TLB и устраняет конфликты ассоциативности кэша.

### **Было:**
```C++
void multiply_optimized(const std::vector<float>& A, const std::vector<float>& B, std::vector<float>& C) {
    constexpr int BLOCK_SIZE = 64;
    // В первых трех бью на блоки по 64 Кб для кешей L1/L2
    #pragma omp parallel for // Базовое расппараллеливание цикла для совместимости с MSVC (Visual Studio)
    for (int i = 0; i < N; i += BLOCK_SIZE) {
        for (int j = 0; j < N; j += BLOCK_SIZE) {
            for (int k = 0; k < N; k += BLOCK_SIZE) {

                for (int ii = i; ii < i + BLOCK_SIZE; ++ii) {

                    // Получаем указатели на начало нужной строки в матрицах C и A
                    float* c_row = &C[ii * N];
                    const float* a_row = &A[ii * N];

                    // Сначала kk, а не jj, чтобы записывать построчно, а не по столбцам
                    for (int kk = k; kk < k + BLOCK_SIZE; ++kk) {

                        float a_ik = a_row[k + kk]; // доступ без умножения

                        // Получаем указатель на нужную строку в матрице B
                        const float* b_row = &B[kk * N];

                        for (int jj = j; jj < j + BLOCK_SIZE; ++jj) {
                            // В итоге линейный сдвиг по памяти и нет умножений адресов
                            c_row[jj] += a_ik * b_row[jj];
                        }
                    }
                }
            }
        }
    }
}
```

#### **Объяснение для себя:**
1. **Почему 6 циклов**
    - Первые три цикла `i, j, k` ограничивают область видимости для остальных трёх `ii, kk, jj`. По исходной матрице происходит передвижение квадратами с длиной 64 элемента.
    - Три внутренних цикла осуществляют математику умножения матриц.

2. **Математика работы с памятью**
   - Выносим более затратное по ресурсам умножение из более глубоких циклов и считаем индекс начала нужной строки один раз (далее только прибавляем к нему):
    ```C++ 
    float* c_row = &C[ii * N];
    const float* a_row = &A[ii * N];
    ```
    - Благодаря слову `const` компилятор сохраняет `a_row` во внутренние регистры и использует повторно не перепроверяя, изменилось ли значение.
    - Строка `float a_ik = a_row[k + kk];` создает локальную переменную, которая нужна в цикле `jj`, чтобы не вычислять одно и то же значение 64 раза. Так же делает возможным **SIMD Broadcasting** (Размножение для векторизации).
    - То же самое с `const float* b_row = &B[kk * N];`.


### **Стало:**
```C++
        ...
            for (int k = 0; k < N; k += BLOCK_SIZE) {
                
                // Создаем непрерывный буфер в L1-кэше для текущего блока матрицы B
                float B_pack[BLOCK_SIZE * BLOCK_SIZE];

                // Копирование блока данных из исходной матрицы в B_pack
                for (int pk = 0; pk < BLOCK_SIZE; ++pk) {
                    const float* b_src_row = &B[(k + pk) * N + j];
                    float* b_dst_row = &B_pack[pk * BLOCK_SIZE];

                    // SIMD-оптимизированное копирование памяти
                    std::copy(b_src_row, b_src_row + BLOCK_SIZE, b_dst_row);
                }

                for (int ii = i; ii < i + BLOCK_SIZE; ++ii) {
                    // Получаем указатели на начало нужной строки в матрицах C и A
                    float* c_row = &C[ii * N]; 
                    const float* a_row = &A[ii * N];

                    // Сначала kk, а не jj, чтобы записывать построчно, а не по столбцам
                    for (int kk = 0; kk < BLOCK_SIZE; ++kk) {

                        float a_ik = a_row[k + kk]; // доступ без умножения

                        // Указатель, который ссылается на начало нужной строки внутри буфера
                        const float* b_row_packed = &B_pack[kk * BLOCK_SIZE];

                        // цикл максимально предсказуем для компилятора, поэтому максимально оптимизирован
                        for (int jj = 0; jj < BLOCK_SIZE; ++jj) {
                            // Скорее всего здесь компилятор применяет инструкцию FMA (слитное умножение-сложение)
                            c_row[j + jj] += a_ik * b_row_packed[jj]; 
                    ...
```

#### **Объяснение для себя:**
Это непосредственно сам Data Packing.
   - **Важная оптимизация:** память выделяется в стеке `float B_pack[BLOCK_SIZE * BLOCK_SIZE];`, что требует лишь 1 такт АЛУ. Генерируется всего одна процессорная инструкция сдвига регистра: `sub rsp, 16384`.
   - **Нюанс:** в памяти лежат мусорные значения от прошлых операций, процессор не тратит время на её заполнение нулями. Именно поэтому сразу после этого мы запускаем цикл `std::copy`, который принудительно перезаписывает этот мусор нашими данными.

**Почему именно std::copy?**
Использование std::copy - это способ получить доступ к аппаратным SIMD-инструкциям копирования памяти. По стандарту для такого типа данных как `float` должно использоваться оптимизированное блочное копирование с максимально широким допустимым регистром (Источник: **cppreference.com** -> Раздел "*Алгоритмы*" -> `std::copy, std::copy_if` -> Глава "*Notes*").

## Почему **мой** результат математически и логически корректен?

### Математическое обоснование нахождения нормы Фробениуса

   - **Источник:** Gene H. Golub, Charles F. Van Loan. *"Matrix Computations"* (4th Edition, Johns Hopkins University Press, 2013).
   - **Где искать:** Глава 2 *Matrix Analysis*, Раздел 2.3 *Vector and Matrix Norms*, подраздел 2.3.2 *The Frobenius Norm*.

В классическом виде из учебника норма Фробениуса для произвольной матрицы $X$ размером $N \times N$ выглядит так:

$$||X||_F = \sqrt{\sum_{i=1}^{N} \sum_{j=1}^{N} |x_{ij}|^2}$$

В коде вычисляется норма не от одной из исходных матриц, а от матрицы, которая является результатом вычитания эталонной матрицы из тестовой.

Пусть $D$ - это матрица разности, где $D = C_{ref} - C_{test}$. Тогда формула принимает следующий вид:

$$||C_{ref} - C_{test}||_F = \sqrt{\sum_{i=1}^{N} \sum_{j=1}^{N} (c_{ref, ij} - c_{test, ij})^2}$$

#### Пометка для себя

   - Именно результат выполнения и является значением, которое математики называют **Нормой Фробениуса** для данной матрицы. Это число показывает физическое "расстояние" между эталонной и тестовой матрицами в многомерном пространстве.

### Реализация нормы Фробениуса в коде
```C++
double check_accuracy(const std::vector<float>& C_ref, const std::vector<float>& C_test) {
    double diff_sq_sum = 0.0;

    // Используем OpenMP для быстрого подсчета суммы квадратов разностей
    // reduction изолирует переменную для каждого потока
    #pragma omp parallel for reduction(+:diff_sq_sum)
    for (int i = 0; i < N * N; ++i) {
        // Обязательно переводим float в double ДО вычитания, 
        // чтобы избежать переполнения при суммировании 16.7 млн элементов
        double diff = static_cast<double>(C_ref[i]) - static_cast<double>(C_test[i]);
        diff_sq_sum += diff * diff;
    }

    return std::sqrt(diff_sq_sum);
}
```
### 0.1 действительно большая погрешность?

   - **Источник:** Nicholas J. Higham. *"Accuracy and Stability of Numerical Algorithms"* (2nd Edition, SIAM, 2002).
   - **Где искать:**  Глава 1, раздел 1.11 *Accumulation of Rounding Errors* и Глава 3 *Matrix Multiplication*. Там описывается статистическая оценка ошибок округления, изначально предложенная Джеймсом Уилкинсоном.

Это фундаментальный академический труд, на который опираются разработчики математических библиотек (включая BLAS и LAPACK) при оценке численной стабильности алгоритмов.

### Формула среднего допустимого отклонения

Для случайных данных размерность $n$ в множителе ошибки заменяется на $\sqrt{n}$. Это фундаментальное свойство дисперсии независимых случайных величин.

Общий вид формулы для **среднего отклонения** в норме Фробениуса:

$$E_{avg} \approx \sqrt{n} \cdot u \cdot |||A| |B|||_F$$

**Разбор параметров задачи:**

1. **$\sqrt{n}$ (Корень из размерности):** У нас $n = 4096$. Корень из $4096$ равен **$64$**.
2. **$u$ (Машинное эпсилон):** Для типа данных `float` предел ошибки одного округления составляет **$\approx 5.96 \cdot 10^{-8}$**.
3. **$|||A| |B|||_F$ (Ожидаемая норма матриц без учета знаков):** Модули равномерно распределенных чисел дают среднее значение $0.5$.

* Математическое ожидание каждого элемента матрицы $|A||B|$ составляет: $4096 \cdot 0.5 \cdot 0.5 = 1024$.
* Норма Фробениуса для такой матрицы: $\sqrt{4096^2 \cdot 1024^2} = 4\,194\,304$.

Подставляем все значения в формулу:

$$E_{avg} \approx 64 \cdot 5.96 \cdot 10^{-8} \cdot 4\,194\,304 \approx 16$$

**Вывод:**
Среднее допустимое отклонение для моих параметров составляет **16**. **0.1** в 160 раз меньше теоретического предела. Возможно это действительно большая погрешность **относительно других** более сложных алгоритмов, с которыми я пока не знаком.

### Тогда может максимальная разность среди элементов выходит за пределы нормы?

   - **Источник:** Nicholas J. Higham, Theo Mary - *"A New Approach to Probabilistic Rounding Error Analysis"* (SIAM Journal on Scientific Computing, Том 41, Выпуск 5, 2019 год).
   - **Где искать:** Раздел 3 *Probabilistic Bounds for Inner Products*, Теорема 3.1.
   - **Суть:** Авторы доказывают, что для скалярного произведения двух векторов длины $N$, средняя вероятностная граница абсолютной ошибки вычисляется так:

$$|fl(x^T y) - x^T y| \le \sqrt{N} \cdot \mathbf{u} \cdot \sum_{k=1}^N |x_k y_k|$$

где $\mathbf{u}$ — машинное эпсилон (для типа `float` это $2^{-24} \approx 5.96 \cdot 10^{-8}$).

**Расчет этой ошибки под задачу:**

* Размер векторов: $N = 4096$.
* Погрешность типа `float`: $\mathbf{u} \approx 5.96 \cdot 10^{-8}$.

Извлекаем корень из $N$ и умножаем на эпсилон:

$$\sqrt{4096} \cdot (5.96 \cdot 10^{-8}) = 64 \cdot (5.96 \cdot 10^{-8}) \approx 3.81 \cdot 10^{-6}$$

Генератор заполнял матрицы числами в диапазоне от **-1.0** до **1.0**:

* Средний **модуль** одного числа: **0.5**.
* Средний модуль их перемножения ($|x_k y_k|$): **0.25**.
* Сумма для 4096 элементов: $4096 \cdot 0.25 = \mathbf{1024}$.

Умножаем коэффициент из первого шага на сумму из второго:

$$3.81 \cdot 10^{-6} \cdot 1024 \approx \mathbf{0.0039}$$

### Реализация поиска максимальной ошибки в коде:
```C++
double check_max_error(const std::vector<float>& C_ref, const std::vector<float>& C_test) {
    double max_diff = 0.0;
    for (size_t i = 0; i < C_ref.size(); ++i) {
        double diff = std::abs(static_cast<double>(C_ref[i]) - static_cast<double>(C_test[i]));
        if (diff > max_diff) {
            max_diff = diff;
        }
    }
    return max_diff;
}
```

**Вывод:**
Математическая модель показывает, что средняя ожидаемая ошибка в задаче составляет **0.0039**. Реальная ошибка в консоли **0.0004**, что почти в 10 раз меньше этого предела. Суть та же: пока что не могу оценить эту погрешность как большую или маленькую, потому что сравнивать не с чем.

### Математическое и архитектурное доказательство предела точности стандарта IEEE 754 **при линейном суммировании**

**Утверждение:** Расхождение $4 \cdot 10^{-4}$ между пользовательской реализацией умножения матриц ($4096 \times 4096$) и библиотекой BLAS является аппаратным пределом стандарта одинарной точности. Улучшение точности **при линейном суммировании** невозможно без катастрофического падения производительности (MFlops) или изменения типа данных.

#### 1. Фундаментальные ограничения формата IEEE 754

Согласно стандарту IEEE 754, тип `float` занимает в памяти 32 бита и имеет следующую структуру:

* 1 бит знака.
* 8 бит экспоненты (определяет масштаб/порядок числа).
* **23 бита мантиссы** (включая 1 неявный старший бит, итого 24 бита).

24 бита мантиссы означают, что процессор аппаратно способен хранить точное значение числа только в пределах $2^{24}$ дискретных шагов. Единичная ошибка округления (машинное эпсилон) для этого типа составляет $\mathbf{u} = 2^{-24} \approx 5.96 \cdot 10^{-8}$. Это гарантирует точность не более чем **7 значащих десятичных цифр**.

#### 2. Механизм потери точности №1: Выравнивание порядков (Swamping)

Для вычисления одной ячейки матрицы процессор выполняет $N = 4096$ последовательных сложений в цикле:

$$S_{i} = S_{i-1} + (a_k \cdot b_k)$$

Операция сложения чисел с плавающей запятой в АЛУ процессора требует равенства их экспонент (порядков). Если к накопленной сумме (например, $S \approx 16.0$) прибавляется новое малое произведение (например, $p \approx 0.25$), АЛУ выполняет следующие действия:

1. Вычисляет разницу порядков: $\Delta = \exp_S - \exp_p$. Для $16.0$ ($2^4$) и $0.25$ ($2^{-2}$) разница составляет $\Delta = 6$.
2. Мантисса меньшего числа сдвигается вправо на $\Delta$ бит, чтобы уравнять масштабы.
3. Биты мантиссы, вышедшие за пределы 24-битного регистра при сдвиге, **безвозвратно удаляются аппаратно**.

Уже при сумме $16.0$ мы теряем $6$ из $24$ бит каждого нового прибавляемого числа. По мере роста аккумулятора младшие биты новых слагаемых физически не доходят до итоговой суммы.

#### 3. Механизм потери точности №2: Катастрофическая потеря значимости (Catastrophic Cancellation)

Поскольку исходные матрицы заполнены случайными числами в диапазоне $[-1.0, 1.0]$, математическое ожидание их произведений равно нулю. Сумма не растет линейно, а представляет собой случайное блуждание, постоянно колеблясь вокруг нуля:


$$16.123456_{float} + (-16.123450_{float}) = 0.000006_{float}$$

В момент такого вычитания старшие (самые точные) биты взаимно уничтожаются. В результате в мантиссе остается "мусор" из младших разрядов, которые до этого уже были искажены эффектом поглощения (Swamping). Чередование этих двух механизмов 4096 раз на каждую ячейку формирует неизбежный аппаратный шум.

#### 4. Архитектурная разница с библиотекой BLAS

Библиотека BLAS выдает немного иной результат не потому, что её алгоритм математически "правильнее", а из-за разницы в использовании аппаратных регистров:

* **Мой код (Sequential Accumulation):** Использует один регистр-аккумулятор для всей цепочки из 4096 сложений. Эффекты выравнивания порядков и вычитания накапливаются максимально.
* **BLAS (Register Blocking / Multiple Accumulators):** На уровне ассемблера микро-ядро BLAS использует несколько (например, 8 или 16) независимых регистров YMM/ZMM. Длинная цепь из 4096 сложений разбивается на параллельные короткие цепи (например, 16 цепей по 256 сложений). В коротких цепях дисперсия разницы экспонент меньше, поэтому отсекается меньше бит. В конце эти регистры суммируются.

Оба подхода являются математически корректными аппроксимациями в пределах 32-битной арифметики. Разница $4 \cdot 10^{-4}$ — это разница между двумя путями аппаратного округления.

## Как так мой код вообще может быть быстрее?

* Библиотека BLAS написана универсально. Она должна уметь умножать матрицы $10 \times 10$, $4096 \times 4096$, $137 \times 512$, транспонированные и нет. В ней куча динамических проверок (if-else) и логики обработки краевых эффектов (когда размер не делится нацело на блок).
Я жестко задал `constexpr int N = 4096;` и `constexpr int BLOCK_SIZE = 64;`. Компилятор знал размер заранее и смог сгенерировать идеально ассемблерный код без лишних проверок.
* Готовые DLL или статики BLAS (например, OpenBLAS из пакета) часто скомпилированы под усредненный процессор, чтобы работать на всех компьютерах без сбоев. Мой код, скомпилированный локально с нужными флагами оптимизации, использует максимум инструкций конкретно моего процессора.
* Существенный вклад внесла упаковка данных, о которой писал выше.


# Другие изменения


## Паралллельная инициализация

Даже если при тестировании кода на обычном процессоре (UMA), правильное распределение страниц памяти при многопоточности - это золотой стандарт HPC. Операционная система физически выделяет память в момент первой записи, а не при вызове `std::vector` (First-Touch Policy).

   - **Источник:** Ruud van der Pas, Eric Stotzer, Christian Terboven - "*Using OpenMP: Portable Shared Memory Parallel Programming*" (MIT Press).
   - **Где искать:** Глава об оптимизации производительности, секция "*NUMA Architectures and First-Touch Allocation*".

   - **Суть:** "*To ensure optimal memory placement, memory should be initialized by the thread that is going to use it... A common pitfall is to initialize data sequentially (e.g., in a master thread) and then process it in parallel.*". Чтобы обеспечить оптимальное размещение в памяти, данные должны инициализироваться тем потоком, который будет их использовать. Частая ошибка: инициализировать данные последовательно, а обрабатывать параллельно.

#### **Было:**
```C++ 
std::fill(C.begin(), C.end(), 0.0f);
start = std::chrono::high_resolution_clock::now();
multiply_optimized(A, B, C);
```

#### **Стало:**
```C++ 
#pragma omp parallel for
    for (int i = 0; i < N * N; ++i) {
        C[i] = 0.0f;
    }
    
    start = std::chrono::high_resolution_clock::now();
    multiply_optimized(A, B, C);
```


## Проблема с #pragma omp parallel for collapse(2)


   - **Источник:** Официальная документация Microsoft Learn (ранее MSDN) для C/C++ разработчиков.
   - **Где искать:** *Справочник по сборке C/C++ (C/C++ Building Reference)* -> *Параметры компилятора MSVC (Compiler Options)* -> Статья **`/openmp` (Enable OpenMP Support)**, Архитектурный обзор **"OpenMP in Visual C++"**.

Microsoft официально заявляет, что стандартный флаг компилятора `/openmp` в Visual Studio реализует спецификацию **OpenMP API версии 2.0** (утвержденную в марте 2002 года).

Директива `collapse(n)` была впервые введена в стандарт **OpenMP 3.0** (в 2008 году). Её алгоритмическая задача - взять `n` вложенных циклов, математически "схлопнуть" их в один большой одномерный цикл и равномерно раздать пулу потоков.

Компиляторы семейства GCC (MinGW) и Clang по умолчанию поддерживают новые стандарты OpenMP (4.5 и выше). Из-за этого код, который компилируется и нормально работает у меня приводит к полному отключению параллелизма циклов у вас.


## Флаг `-ffast-math` ухудшает точность?


#### **Для компилятора MSVC (Visual Studio):**

   - **Источник:** Официальная документация Microsoft Learn.
   - **Где искать:** *Справочник по сборке C/C++ (C/C++ Building Reference)* -> *Параметры компилятора MSVC (Compiler Options)* -> Статья **`/fp` (Specify floating-point behavior)**.

Параметр `/fp:fast` отключает строгую математическую семантику, разрешая компилятору переупорядочивать выражения и объединять операции (например, генерировать инструкции FMA), что приводит к потере точной воспроизводимости операций с плавающей точкой.

#### **Для компиляторов семейства GCC (MinGW):**

   - **Источник:** Официальная документация GCC (GCC Online Documentation).
   - **Где искать:** *Options That Control Optimization* -> Статья **`-ffast-math`**.

Данный флаг нарушает строгую стандартизацию IEEE/ISO для математических функций, отключая гарантии ассоциативности и проверки на специальные значения (NaN, Infinity).

### Плюсы и минусы использования (`-ffast-math` / `/fp:fast`)

**Плюсы:**

   - Компилятор получает право упаковывать циклы вычислений с плавающей точкой в широкие векторные регистры (AVX2/AVX-512), обрабатывая по 8–16 чисел за раз.
   - Компилятор заменяет отдельные инструкции умножения и сложения на одну аппаратную инструкцию (умножил и сразу прибавил). Это удваивает пиковую вычислительную мощность (FLOPS) алгоритма.
   - Для алгоритмов линейной алгебры (таких как умножение матриц $4096 \times 4096$) снятие жестких ограничений стандарта дает ускорение в 2–8 раз в зависимости от процессора.

**Минусы:**

   - Для чисел с плавающей точкой (a + b) + c не всегда равно a + (b + c). Из-за того, что флаг перегруппировывает порядок операций ради скорости, итоговый результат будет накапливать микроскопическую погрешность в младших битах (именно поэтому норма разности не нулевая).
   - Результат вычислений одной и той же программы может отличаться (на уровне миллионных долей) на разных компьютерах, разных версиях компилятора или при разном размере блока (Tiling).
   - Если в матрицах случайно возникнет деление на ноль или переполнение (Infinity), программа может выдать некорректный мусорный результат без предупреждения, так как безопасные математические проверки отключены.


## Почему команда `#pragma omp simd` у вас не влияла на скорость, а у меня ухудшала?


В Visual Studio встроенным (если не выбрать другой вручную) является компилятор **MSVC**. В VS Code встроенных нет. Я выбрал **MinGW** для удобной работы с CMake. Но команды для этих компиляторов разные. Ваш просто не мог прочитать эти инструкции и не выполнял их.

### **В моём случае возможны следующие причины:**

1. **Конфликт с автовекторизацией (отключение эвристик компилятора)**

   - **Источник:** Barbara Chapman, Gabriele Jost, Ruud van der Pas - *"Using OpenMP: Portable Shared Memory Parallel Programming"* (MIT Press), а также официальная документация к компиляторам LLVM/Clang.
   - **Где искать:** Разделы, посвященные `SIMD` директивам (появились в OpenMP 4.0), и статьи разработчиков LLVM о реализации OpenMP.

   - **Суть:** В документации разработчиков компиляторов подчеркивается, что `#pragma omp simd` - это императивная команда. Вы заставляете компилятор использовать конкретный механизм OpenMP. Это часто отключает встроенный векторизатор компилятора (Loop Vectorizer), который умеет применять более продвинутые паттерны (например, автоматическое слияние инструкций в FMA или multi-versioning), опираясь на архитектуру целевого процессора.

2. **Поломка "Модели стоимости" (Cost Model) и развертывания циклов**

   - **Источник:** Agner Fog — *"Optimizing Software in C++"* (Признанный золотой стандарт по низкоуровневой оптимизации для x86/x64).
   - **Где искать:** Глава 7 (*Optimizations in C++*), подразделы *Loop unrolling* и *Vectorization*.

   - **Суть:** Агнер Фог детально объясняет работу модели стоимости компиляторов. Он пишет: "*Compilers will usually unroll a loop automatically if this appears to be profitable. The programmer does not have to unroll a loop manually...*". Если ставить прагму на короткий внутренний цикл (BLOCK_SIZE = 64), это жесткое указание векторизовать его. Компилятору приходится подчиниться, хотя его внутренняя модель стоимости могла посчитать, что для такого короткого участка гораздо выгоднее сделать *Loop Unrolling* (полное развертывание) без накладных расходов на векторные проверки.

3. **Проблема выравнивания памяти (Memory Alignment)**

   - **Источник:** Randal E. Bryant, David R. O'Hallaron — "*Computer Systems: A Programmer's Perspective*" (CS:APP). Одно из главных пособий по системному программированию в мировых университетах.
   - **Где искать:** Глава 5 (*Optimizing Program Performance*), разделы про архитектуру процессоров, SIMD (SSE/AVX) и обращения к памяти.

   - **Суть:** Авторы подробно разбирают падение скорости работы программы (потеря тактов процессора) из-за невыровненного доступа к памяти при использовании векторных регистров. Без директив умный компилятор использует технику *loop peeling* — он делает пару скалярных вычислений, пока указатель не выровняется по границе 32 байт, а затем запускает быстрый векторизованный код. Принудительная директива SIMD часто заставляет компилятор сгенерировать инструкции невыровненного доступа (например, `vmovups` вместо быстрого `vmovaps`), что приводит к деградации производительности на уровне кэша L1.
